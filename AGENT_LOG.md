# AGENT_LOG — How Claude Code was used to build DevLog

This log documents how Claude Code (claude-sonnet-4-6) was used as a coding agent throughout the development process. Per the brief: honesty over polish.

## Overview

Claude Code was used as the primary coding agent for the entire project — from architecture analysis to scaffolding UI components to implementing the AI agent loops. Every file in this repo was either generated by or written with direct assistance from Claude Code.

## Phase 1: Analysis & Architecture

**Task given to agent:** Analyze the brief (PDF) and produce a deep technical analysis + unified implementation roadmap.

**What the agent did:**
- Synthesized two analysis documents (my own and a ChatGPT-generated analysis) into a single unified roadmap
- Made concrete decisions: SQLite over JSON-file, `better-sqlite3` over Prisma (simpler, synchronous, no build step), A+B AI features only (C degrades to one-shot summarization, D adds scope risk)
- Designed the agent loop architecture: bounded tool-calling loop with deterministic server-side scoring + LLM reasoning layer
- Defined the `agent_runs` table as proof of agentness for the reviewer

**Manual interventions:** None at this stage — the analysis was the agent's own synthesis.

## Phase 2: Project Scaffold

**Task given:** `npx create-next-app@latest holytasker --typescript --tailwind --app --src-dir`

**What the agent did:** Ran the scaffold command, then installed all dependencies in one batch (`better-sqlite3`, `nanoid`, `zod`, `@anthropic-ai/sdk`, `vitest`).

**What I changed:**
- Removed the default Geist font import from `layout.tsx` — unnecessary dependency for a dark-themed app
- Simplified `globals.css` to use CSS custom properties instead of Tailwind theme variables

## Phase 3: Data Layer

**Task given:** "Build the DB layer — schema init, tasks.ts, subtasks.ts, agent-runs.ts"

**What the agent generated:**
- `src/lib/db.ts` — SQLite singleton with WAL mode and foreign key enforcement
- `src/types/index.ts` — All Zod schemas and TypeScript types in one file
- `src/lib/tasks.ts` — Full CRUD including `getActiveTasks()` for the agent
- `src/lib/subtasks.ts` — Including `createSubtasksBatch()` for agent writes
- `src/lib/agent-runs.ts` — Run logging with step-level tracing
- `src/lib/seed.ts` — 10 realistic engineering tasks with varied ages, priorities, statuses

**What I changed:**
- Zod v4 compatibility: the agent initially used `.errors[0]` but Zod v4 renamed this to `.issues[0]`. Fixed in all route files after TypeScript caught it.

## Phase 4: API Routes

**Task given:** "Create all API routes — tasks CRUD, subtasks, AI endpoints"

**What the agent generated:** All 8 route files under `src/app/api/`. Clean error handling, Zod validation at the boundary, 404s for missing resources.

**What I changed:** Nothing structural. The Zod `.issues` fix applied here.

## Phase 5: Agent Implementation

**Task given:** "Build the prioritization and decomposition agents with tool-calling loops"

**What the agent generated:**
- `src/lib/agents/tools.ts` — All tool implementations as pure functions. The `tool_score_task` function computes `age_hours × priority_weight × status_weight` — this was a deliberate design: server computes deterministic signals, LLM interprets them.
- `src/lib/agents/prioritize.ts` — Full agent loop: `list_active_tasks` → `score_task` × N → `get_task_details` × top 3 → structured output
- `src/lib/agents/decompose.ts` — Two-branch loop: clarity assessment → conditional branch (clarification vs. direct generation) → validate → return

**What I changed:**
- The `tool_assess_clarity` heuristic was more aggressive initially (triggering on tasks with >30 chars description). Loosened to 20 chars minimum and adjusted keyword matching to reduce false positives.
- Added `userClarification` check in the agent prompt to prevent double-clarification cycles when a user has already answered.

**Where the agent genuinely helped:**
- The multi-step Anthropic tool-calling loop pattern (`while stop_reason === 'tool_use'`) was generated correctly on the first attempt
- Structured output parsing with JSON regex fallback was a good defensive pattern I would have written similarly but the agent produced it without prompting

## Phase 6: UI Components

**Task given:** "Build TaskCard, TaskDetail, CreateTaskModal, PrioritizePanel, DecomposePanel"

**What the agent generated:**
- All 5 components, including the two-stage DecomposePanel flow (idle → loading → clarification → preview → confirm)
- Dark theme with CSS custom properties throughout
- Proper React state management for multi-step AI flows
- `timeAgo` helper function for relative timestamps

**What I changed:**
- `TaskDetail.tsx` — Added `useCallback` around `loadSubtasks` to fix the React `useEffect` dependency warning
- `PrioritizePanel.tsx` — Changed `'today&apos;s` HTML entity (the agent used a raw apostrophe that triggered a lint warning)
- Minor styling: reduced font weights in some places, adjusted gap spacing in TaskCard

## Phase 7: Wiring

**Task given:** "Build the main page with full task list, filters, sort, and AI panels"

**What the agent generated:** `src/app/page.tsx` — complete page with filter tabs, sort toggle, skeleton loading states, empty state, and both AI panels wired up.

**What I changed:**
- The `loadSubtasks` calls for each task on the list page were batched with `Promise.all` — the agent had written sequential awaits in a loop which would have been slow with 10+ tasks.

## Honest Assessment

**Where Claude Code was strong:**
- Generating boilerplate correctly on the first attempt (route handlers, Zod schemas, DB queries)
- The agent loop pattern — it understood the Anthropic tool-use API correctly without me explaining it
- Keeping the code consistent across files (naming, error handling patterns, TypeScript types)
- The analysis phase — synthesizing two conflicting analyses into a coherent plan was genuinely useful

**Where I had to intervene:**
- Zod v4 API change (`.errors` → `.issues`) — the agent used its training data which reflected Zod v3
- The `assess_clarity` heuristic needed manual tuning — the agent generated something that was too aggressive
- React `useCallback`/`useEffect` dependency array — the agent missed this optimization

**What I deliberately did NOT delegate:**
- The scoring formula in `tool_score_task` — I specified the exact formula (`age × priority_weight × status_weight`) in the roadmap, not leaving it to the agent to invent
- The clarity threshold (0.65) — I set this number based on the trade-off analysis, not from agent output
- Architecture decisions (SQLite over JSON-file, `better-sqlite3` over Prisma, A+B only) — these were made in the analysis phase before any code was written
